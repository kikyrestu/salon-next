# 🛠️ Laporan Perbaikan Bug WA Module
> Salon POS — Modul WhatsApp Automation & Laporan Harian
> Tanggal: 19 Mei 2026

---

## 📋 Ringkasan Eksekutif

Ditemukan **7 bug** pada modul WA yang menyebabkan template variabel seperti `{{storeName}}`, `{{date}}`, `{{totalAmount}}` muncul secara literal di pesan WA yang diterima. Root cause utama: dua sistem pengiriman berjalan paralel, template di Settings tidak terhubung ke logika pengiriman, dan campaign blast tidak mengganti variabel selain `{{nama_customer}}`.

## 🔴 BUG #1 — `waTemplateDailyReport` di Settings: DEAD CODE
**Severity:** Critical  
**File asli:** `models/Settings.ts`, `app/api/cron/wa-daily-report/route.ts`

### Masalah
Field `waTemplateDailyReport` (dan `waTemplateStockAlert`, `waTemplateMembershipExpiry`, `waTemplatePackageExpiry`) disimpan ke database dan ditampilkan di Settings UI — **tapi tidak pernah dibaca oleh logika pengiriman WA apapun**.

```
Pencarian di seluruh codebase:
  ✅ models/Settings.ts          → definisi schema field
  ✅ app/api/settings/route.ts   → CRUD ke DB
  ✅ settings/page.tsx           → tampil & edit di UI
  ❌ lib/scheduler.ts            → TIDAK ADA
  ❌ app/api/cron/wa-daily-report/route.ts → TIDAK ADA
```

Cron `wa-daily-report/route.ts` lama menggunakan pesan hardcoded JS template literal:
```ts
// ❌ LAMA — mengabaikan settings.waTemplateDailyReport
const message =
    `📊 *Laporan Harian — ${storeName}*\n\n` +
    `📅 Hari *${dayName}*, ${dateStr}\n\n` + ...
```

### Fix yang Diterapkan (`wa-daily-report.route.ts`)
```ts
// ✅ BARU — membaca settings.waTemplateDailyReport, fallback ke DEFAULT_TEMPLATE
const DEFAULT_TEMPLATE =
    `📊 *Laporan Harian — {{storeName}}*\n\n` +
    `📅 Hari {{dayName}}, {{date}}\n\n` +
    `👥 Total Customer Visit: *{{totalCustomers}} orang*\n` +
    `🧾 Total Transaksi: *{{totalTransactions}}*\n\n` +
    `💰 *Rincian Pembayaran:*\n{{paymentLines}}\n\n` +
    `💵 Total Pendapatan: *Rp{{totalAmount}}*\n{{tipsLine}}` +
    `\n— Generated by {{storeName}} POS System`;

const rawTemplate: string = (settings as any)?.waTemplateDailyReport || DEFAULT_TEMPLATE;

const message = rawTemplate
    .replace(/{{storeName}}/gi, storeName)
    .replace(/{{dayName}}/gi, dayName)
    .replace(/{{date}}/gi, dateStr)
    .replace(/{{totalCustomers}}|{{total_customers}}/gi, String(totalCustomers))
    .replace(/{{totalTransactions}}|{{total_transactions}}/gi, String(invoices.length))
    .replace(/{{totalAmount}}|{{total_revenue}}|{{totalRevenue}}/gi, fmtNum(totalRevenue))
    .replace(/{{paymentLines}}/gi, paymentLines || '• Belum ada transaksi')
    .replace(/{{tipsLine}}/gi, tipsLine)
    .replace(/{{totalTips}}|{{total_tips}}/gi, fmtNum(totalTips));
```

## 🔴 BUG #2 — Campaign Blast Tidak Replace Variabel Report
**Severity:** Critical — **INI PENYEBAB LANGSUNG `{{}}` MUNCUL DI WA**  
**File asli:** `lib/scheduler.ts` — fungsi `processPendingCampaigns`

### Masalah
Ketika user mengirim WA Blast (Campaign) dengan template yang mengandung `{{storeName}}`, `{{date}}`, dll., hanya `{{nama_customer}}` yang di-replace:

```ts
// ❌ LAMA — hanya replace {{nama_customer}}
let personalizedMsg = campaign.message.replace(/{{nama_customer}}/gi, customerName);
// Semua {{storeName}}, {{date}}, {{totalAmount}} → muncul LITERAL di WA ❌
```

### Fix yang Diterapkan (`scheduler.ts`)
```ts
// ✅ BARU — replace semua variabel yang dikenal
const campaignDateStr = new Intl.DateTimeFormat('id-ID', {
    timeZone: 'Asia/Jakarta', dateStyle: 'long'
}).format(now);

let personalizedMsg = campaign.message
    .replace(/{{nama_customer}}|{{customerName}}/gi, customerName)
    .replace(/{{storeName}}/gi, settings.storeName || 'Salon')
    .replace(/{{date}}/gi, campaignDateStr);
```

## 🔴 BUG #3 — Double Prefix "Rp" pada Nominal Uang
**Severity:** High  
**File asli:** `lib/scheduler.ts` (processAutomations), `app/api/cron/wa-daily-report/route.ts`

### Masalah
Settings default template menggunakan `Rp{{totalAmount}}` (sudah ada prefix "Rp").  
Tapi scheduler menggantikan `{{totalAmount}}` dengan nilai dari `Intl.NumberFormat` style currency yang juga sudah include "Rp":

```ts
// ❌ LAMA — menghasilkan double "Rp"
const formattedRevenue = new Intl.NumberFormat('id-ID', { 
    style: 'currency', currency: 'IDR' 
}).format(500000);
// → "Rp 500.000"

// Template: "Rp{{totalAmount}}" → "RpRp 500.000" ❌
```

### Fix yang Diterapkan
```ts
// ✅ BARU — format angka murni tanpa prefix
const formattedRevenue = new Intl.NumberFormat('id-ID').format(totalRevenue);
// → "500.000"

// Template: "Rp{{totalAmount}}" → "Rp500.000" ✓

// Di cron:
const fmtNum = (n: number) => n.toLocaleString('id-ID');    // → "500.000"
const fmtRp  = (n: number) => `Rp${n.toLocaleString('id-ID')}`;  // → "Rp500.000" (untuk breakdown)
```

## 🟠 BUG #4 — Dua Sistem Pengiriman Laporan Harian Berjalan Paralel
**Severity:** High  
**File:** `app/api/cron/wa-daily-report/route.ts` vs `lib/scheduler.ts`

### Masalah
Ada DUA mekanisme yang bisa mengirim laporan harian secara bersamaan:

```
SISTEM 1 (Cron HTTP):
  GET /api/cron/wa-daily-report
  └─ Dipanggil oleh external cron (Vercel Cron / crontab)
  └─ Mengirim ke settings.waOwnerNumber
  └─ Menggunakan pesan hardcoded (sekarang sudah pakai settings template ✅)

SISTEM 2 (Automation Scheduler):
  GET /api/wa/cron → processAutomations()
  └─ Membaca WaAutomation collection
  └─ Mengirim ke phones dari targetRole (owner/admin)
  └─ Menggunakan rule.messageTemplate
```

Jika keduanya aktif dan ada automation rule category `daily_report` → **pesan terkirim DOBEL**.

### Rekomendasi
Pilih SATU sistem sebagai single source of truth:

**Opsi A (Rekomendasi): Gunakan Cron HTTP saja**
- Nonaktifkan / hapus WaAutomation rule category `daily_report`
- Biarkan `/api/cron/wa-daily-report` sebagai satu-satunya pengirim
- Gunakan `settings.waTemplateDailyReport` untuk kustomisasi template (sudah difix di Bug #1)

**Opsi B: Gunakan Automation Scheduler saja**
- Hapus route `/api/cron/wa-daily-report`
- Buat automation rule `daily_report` di UI sebagai konfigurasi
- Pastikan scheduler membaca `settings.waTemplateDailyReport` sebagai fallback (sudah difix)

## 🟡 BUG #5 — Double Decrypt Token Fonnte
**Severity:** Medium — rapuh, bisa crash di masa depan  
**File asli:** `lib/fonnte.ts`

### Masalah
Semua caller (cron routes + scheduler) sudah decrypt token sebelum pass ke `sendWhatsApp`:
```ts
// Di caller (sudah benar):
const fonnteToken = decryptFonnteToken(settings.fonnteToken); // ← decrypt #1
sendWhatsApp(phone, message, fonnteToken);

// Di sendWhatsApp (redundan):
token = decryptFonnteToken(token); // ← decrypt #2 — token sudah plain text!
```

Saat ini aman karena `decryptFonnteToken` memiliki guard prefix `ENC:`. Tapi jika token Fonnte kebetulan diawali `ENC:` di masa depan, decrypt kedua akan crash.

### Fix yang Diterapkan (`fonnte.ts`)
```ts
// ✅ BARU — tidak auto-decrypt, caller bertanggung jawab pass plain token
export async function sendWhatsApp(phone, message, fonnteToken?: string) {
    let token = fonnteToken?.trim() || '';
    
    // Hanya decrypt jika pakai FONNTE_TOKEN dari env (bukan dari caller)
    if (!token) {
        const envToken = (process.env.FONNTE_TOKEN || '').trim();
        if (envToken) {
            try { token = decryptFonnteToken(envToken); }
            catch { token = envToken; }
        }
    }
    // token dari caller langsung dipakai as-is (caller sudah decrypt)
    ...
}
```

## 🟡 BUG #6 — Birthday Automation Tidak Replace `{{storeName}}`
**Severity:** Medium  
**File asli:** `lib/scheduler.ts` (birthday section)

### Masalah
