## 🐛 BUG #1 — Double Scheduler Init + Layout.tsx Tanpa Runtime Guard

### Deskripsi

Ada **dua tempat** yang memanggil `startWaScheduler()`:

**File 1:** `instrumentation.ts` ← BENAR, ada runtime guard
```ts
export async function register() {
    if (process.env.NEXT_RUNTIME === 'nodejs') { // ← ada guard ✓
        const { startWaScheduler } = await import('./lib/scheduler');
        startWaScheduler();
    }
}
```

**File 2:** `app/layout.tsx` ← BERMASALAH, tidak ada runtime guard
```ts
import { startWaScheduler } from "@/lib/scheduler";

startWaScheduler(); // ← module-level call, tanpa NEXT_RUNTIME check ❌
```

### Mengapa Ini Bermasalah?

`app/layout.tsx` adalah React Server Component. Kode module-level di sini dieksekusi saat **module initialization**, bukan hanya saat server start. Tanpa pengecekan `process.env.NEXT_RUNTIME === 'nodejs'`, `startWaScheduler()` bisa dieksekusi di konteks yang salah.

Yang lebih kritis: `instrumentation.ts` menggunakan `NEXT_RUNTIME === 'nodejs'` sebagai guard, tapi kalau ada race condition atau environment issue yang membuat instrumen ini tidak berjalan lebih dulu, `schedulerStarted` flag tetap `false`. Lalu layout.tsx jadi fallback — tapi tanpa runtime check, ini bisa gagal secara silent (misalnya `node-cron` tidak tersedia di worker tertentu) sehingga scheduler **tidak pernah benar-benar aktif**.

**Akibat langsung:** `processPendingWaSchedules()` tidak pernah dipanggil → semua WaSchedule tetap `pending` selamanya.

### Fix #1 — Hapus `startWaScheduler()` dari `layout.tsx`

**File: `app/layout.tsx`**

```diff
- import { startWaScheduler } from "@/lib/scheduler";
-
- startWaScheduler();
-
  import type { Metadata } from "next";
  import { Geist, Geist_Mono } from "next/font/google";
  import "./globals.css";
  import SessionProviderWrapper from "@/components/providers/SessionProviderWrapper";
  import { SettingsProvider } from "@/components/providers/SettingsProvider";

  // (tidak ada lagi import scheduler di sini)
```

Serahkan sepenuhnya ke `instrumentation.ts` yang sudah benar. Satu titik startup, satu titik kontrol.

## 🐛 BUG #2 (PALING KRITIS) — Frontend Mismatch Response Field: "Blast Sent Successfully!" Palsu

### Deskripsi

Ini adalah **root cause utama** kenapa user melihat "Blast Sent Successfully!" tapi tidak ada WA yang datang.

Setelah refactor blast dari "kirim langsung" ke "queue-based", API response berubah:

**API Response LAMA (direct send):**
```json
{
  "success": true,
  "total": 50,
  "sent": 48,
  "failed": 2,
  "failedRecipients": [...]
}
```

**API Response BARU (queue-based), `POST /api/wa/blast-targets`:**
```json
{
  "success": true,
  "queued": true,
  "message": "50 pesan dijadwalkan. Akan dikirim otomatis dalam beberapa menit.",
  "campaignId": "abc123xyz",
  "targetCount": 50
}
```

**Masalah:** Frontend di `wa-marketing/page.tsx` **tidak diupdate** dan masih mengakses field lama:

```tsx
// wa-marketing/page.tsx — kondisi saat ini (BERMASALAH)
{blastResult.failed > 0 ? (   // undefined > 0 = FALSE → selalu false!
  <span>Partially Sent</span>
) : (
  <span>Blast Sent Successfully!</span>  // ← selalu render ini
)}
<p>{blastResult.total}</p>   // → undefined → blank
<p>{blastResult.sent}</p>    // → undefined → blank  
<p>{blastResult.failed}</p>  // → undefined → blank
```

Karena `undefined > 0` selalu `false`, frontend **selalu** render "Blast Sent Successfully!" dengan semua stats kosong. User mengira WA sudah terkirim, padahal blast hanya masuk antrian dan menunggu scheduler.

### Fix #2a — Update handleBlast Result UI

Ubah bagian `{blastResult && (...)}` di `wa-marketing/page.tsx`:

```tsx
{/* Blast result — setelah blast berhasil masuk queue */}
{blastResult && (
  <div className="bg-white rounded-xl border border-blue-200 shadow-sm overflow-hidden">
    <div className="px-6 py-4 bg-blue-50 border-b border-blue-100">

            ? "Campaign Berhasil Dijadwalkan!"

        </span>
      </h2>
    </div>
    <div className="p-6 grid grid-cols-3 gap-4">
      <div className="bg-gray-50 rounded-lg p-3 text-center">
        <p className="text-2xl font-bold text-gray-900">
          {blastResult.queued
            ? blastResult.targetCount   // ← gunakan targetCount untuk queue
            : blastResult.total}
        </p>
        <p className="text-xs text-gray-500 font-medium">Total Target</p>
      </div>
      {blastResult.queued ? (
        // Mode queue: tampilkan info antrian
        <div className="col-span-2 bg-blue-50 rounded-lg p-3 flex items-center gap-3">
          <div>
            <p className="text-sm font-semibold text-blue-800">
              Pesan masuk antrian ✓
            </p>
            <p className="text-xs text-blue-600 mt-1">
              Akan dikirim otomatis oleh scheduler dalam beberapa menit.
              Pantau progress di tab "Antrian Campaign".
            </p>
            {blastResult.campaignId && (
              <p className="text-xs text-blue-400 mt-1 font-mono">
                ID: {blastResult.campaignId}
              </p>
            )}
          </div>
        </div>
      ) : (
        // Mode direct send (fallback lama): tampilkan sent/failed
        <>
          <div className="bg-green-50 rounded-lg p-3 text-center">
            <p className="text-2xl font-bold text-green-700">
              {blastResult.sent}
            </p>
            <p className="text-xs text-gray-500 font-medium">Terkirim</p>
          </div>
          <div className={`${blastResult.failed > 0 ? "bg-red-50" : "bg-gray-50"} rounded-lg p-3 text-center`}>
            <p className={`text-2xl font-bold ${blastResult.failed > 0 ? "text-red-700" : "text-gray-400"}`}>
              {blastResult.failed}
            </p>
            <p className={`text-xs font-medium ${blastResult.failed > 0 ? "text-red-600" : "text-gray-400"}`}>
              Gagal
            </p>
          </div>
        </>
      )}
    </div>
    {/* Tampilkan content warnings jika ada */}
    {blastResult.contentWarnings && blastResult.contentWarnings.length > 0 && (
      <div className="px-6 pb-4">
        <div className="bg-amber-50 border border-amber-200 rounded-lg p-3">
          <p className="text-xs font-semibold text-amber-800 mb-1">⚠️ Peringatan Konten:</p>
          {blastResult.contentWarnings.map((w: string, i: number) => (
            <p key={i} className="text-xs text-amber-700">• {w}</p>
          ))}
        </div>
      </div>
    )}
  </div>
)}
```

## 🐛 BUG #3 — WA Schedule "Menunggu Terus": Akibat Langsung Bug #1

### Deskripsi

`processPendingWaSchedules()` di `scheduler.ts` mencari:

```ts
await WaSchedule.find({
    scheduledAt: { $lte: now },
    status: 'pending',
})
```

Query ini benar. Masalahnya adalah fungsi ini **tidak pernah dipanggil** karena scheduler tidak berhasil start (Bug #1).

Selain itu, ada satu potensi silent failure: kalau `template?.message` null/undefined, schedule langsung ditandai `failed` tanpa log yang jelas:

```ts
if (!template?.message) {
    await WaSchedule.findByIdAndUpdate(schedule._id, { status: 'failed' });
    totalFailed += 1;
    continue; // silent, tidak ada error message
}
```

### Fix #3 — Tambahkan Log Informatif + Tombol Manual Trigger di UI

Di `scheduler.ts`, tambahkan log yang informatif:

```ts
// Di processPendingWaSchedules(), ganti silent fail dengan log:
if (!template?.message) {
    console.error(`[WaSchedule] Schedule ${schedule._id}: template tidak ditemukan atau message kosong. templateId: ${schedule.templateId}`);
    await WaSchedule.findByIdAndUpdate(schedule._id, { status: 'failed' });
    totalFailed += 1;
    continue;
}
```

Dan di UI (misalnya di halaman WA Templates atau Settings), tambahkan tombol manual trigger:

```tsx
// Tombol debug/manual trigger — bisa ditaruh di settings halaman WA
const triggerScheduler = async () => {
  const res = await fetch("/api/wa/trigger", {
    method: "POST",
    headers: {
      "x-store-slug": slug,
      "Content-Type": "application/json",
    },
  });
  const data = await res.json();
  if (data.success) {
    alert(`Scheduler triggered. WaSchedule: total=${data.data?.total}, sent=${data.data?.sent}, failed=${data.data?.failed}`);
    fetchQueue(); // refresh antrian
  } else {
    alert("Trigger gagal: " + data.error);
  }
};

// Render:
<button
  onClick={triggerScheduler}
  className="px-3 py-1 text-xs bg-gray-100 hover:bg-gray-200 rounded border text-gray-600"
>
  🔁 Manual Trigger Scheduler
</button>
```

> **Note:** Tombol ini aman — endpoint `/api/wa/trigger` sudah ada RBAC check. Gunakan hanya untuk testing/debug, atau saat server baru restart dan ada schedule yang tertunda.

## ⚠️ Catatan Penting Lainnya

### Jam Operasional Scheduler (Default 08:00 - 20:00 WIB)

`processPendingCampaigns()` hanya berjalan di jam operasional:
```ts
const opStart = settings.waOperationalHoursStart ?? 8;
const opEnd = settings.waOperationalHoursEnd ?? 20;
if (hourWIB < opStart || hourWIB >= opEnd) continue;
```

**Kalau kamu blast di luar jam ini → campaign tidak akan diproses!** Pastikan cron tetap aktif dan blast dilakukan di jam 08:00-20:00 WIB.

### Kecepatan Pengiriman: 2 Pesan / Menit

```ts
const maxPerTick = Math.min(2, remainingQuota); // 2 pesan per menit
// + delay 20-40 detik per pesan (anti-spam detection)
```

Untuk 100 penerima: butuh ~50 menit sampai semua terkirim. Ini normal dan by design untuk menghindari ban dari WA.

### Daily Limit Berdasarkan Umur Device

```ts
if (ageDays <= 7) dailyLimit = 10;
else if (ageDays <= 14) dailyLimit = 20;
else if (ageDays <= 30) dailyLimit = 50;
else dailyLimit = 100;
```

Jika device Fonnte baru (<7 hari), blast hanya 10 pesan/hari. Kalau blast ke 50 orang, sisanya tidak terkirim sampai hari berikutnya.
